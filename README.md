## Engineering Style

- I start with the business decision, then define the entity grain, target, label timing, features available at prediction time, and evaluation metric.
- I prefer simple baselines and clear evidence before complex models.
- I keep AI outputs controlled: deterministic fields stay separate from generated explanation text, and sensitive recommendations need citations, thresholds, approvals, monitoring, and human accountability.
- I think about failure modes early: leakage, stale features, unsafe SQL, PII exposure, model drift, brittle prompts, missing approvals, weak audit trails, and unclear rollback paths.
- I design data systems as products: contracts, tests, lineage, observability, ownership, documentation, and operator workflows are part of the system, not afterthoughts.
- I make repos easy for a team lead to review: clear architecture, reproducible commands, local-first demos, and visible production trade-offs.
